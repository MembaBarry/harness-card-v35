# AI Report Audit Checklist

Use this checklist before accepting any AI-generated research report into Harness Card V3 project direction.

AI research reports are not project canon. They are raw source material until reviewed.

## Purpose

This checklist protects the project from polished but unsupported AI analysis.

It is designed to separate:

- grounded evidence
- plausible but unverified claims
- unsupported claims
- false or contradicted claims
- useful recommendations
- decorative theory

## 1. Claim Type

For every major claim, mark one:

- **Verified** — source exists and supports the claim.
- **Plausible but unverified** — may be true, but needs checking.
- **Unsupported** — stated confidently without evidence.
- **False or contradicted** — evidence points against it.
- **Opinion / recommendation** — useful only as judgment, not fact.

Do not treat unsupported claims as evidence.

## 2. Source Check

For each factual claim, ask:

- Is there a source?
- Is the source named?
- Is there a working link?
- Does the source actually support the claim?
- Is the claim stronger than the source allows?

If there is no source, quarantine the claim.

## 3. Competitor Check

For every named competitor or adjacent product, ask:

- Does it exist?
- Does it do what the report claims?
- Is it actually adjacent to Harness Card V3?
- Is it a direct threat, indirect threat, or only vaguely related?
- Is the comparison based on current information?
- Is the source column filled with real, checkable sources?

Do not accept competitor tables with blank source columns.

## 4. Naming / SEO Check

For naming claims, ask:

- Was a real search performed?
- What exact terms were searched?
- What showed up on page one?
- Are collisions actually relevant to this product?
- Are the collisions survivable with subtitle, README wording, or repo metadata?
- Is the SEO concern proven, plausible, or exaggerated?

Do not rebrand based on unsourced SEO panic.

## 5. Academic / Technical Citation Check

For every academic or technical claim, ask:

- Is the author correct?
- Is the year correct?
- Is the title provided?
- Is the venue, DOI, arXiv page, or publisher provided?
- Does the paper actually say what the report claims?
- Is the report using the paper as evidence or just decoration?

If a citation cannot be verified, cut it or quarantine it.

## 6. Hype Detector

Flag language like:

- proves
- guarantees
- universal
- no competitor
- deterministic
- empirical evaluations show
- realigns attention mechanism
- closed-loop runtime controller
- mathematically validates

These phrases require strong evidence.

Without evidence, downgrade them to hypothesis or remove them.

## 7. Actionability Test

For each recommendation, ask:

- Can this be acted on today?
- Does it reduce user friction?
- Does it preserve local-first simplicity?
- Does it improve evidence quality?
- Does it avoid overbuilding?
- Does it help the next build step?

If not, put it in the parking lot.

## 8. Project Canon Rule

A claim becomes accepted project direction only after:

1. human review,
2. source verification,
3. alignment with project constraints,
4. usefulness to the next build step.

Until then, it remains source material, not canon.

- Use **AI chat circuit breaker** as the clearest public-facing phrase.
- Do not hard-rebrand yet.

- Keep the product local-first, free, and simple.
- Rebuild competitive intelligence with real sources.
- Separate evidence from assumptions before making public claims.

## 10. Current Strongest Safe Claim

## 11. What This Checklist Prevents

This checklist exists to prevent:

- fake competitor certainty
- fake academic authority
- premature rebranding
- unsupported novelty claims
- inflated SEO panic
- overbuilt product direction
- AI-generated research becoming canon by accident

Build the parser, not the mythology.
